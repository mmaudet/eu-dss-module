# eu-dss — Increment B: cross-platform browser access (design)

- **Date:** 2026-05-30
- **Status:** design APPROVED (strategy chosen, Windows MSI added)
- **Scope:** Make signing work from a browser by serving the local agent over HTTPS (self-signed, accept-once), provide cross-OS launch, and a **Windows MSI installer** (priority demo target). Increment C (multi-user/hosting) is out of scope.

## 1. Goal & context

After increment A, the document-signing core works and validation runs end-to-end from the browser. The **sign** flow from a browser is blocked: a hosted **HTTPS** site cannot call the agent at `http://localhost:9795` (mixed content / Private Network restrictions), and even in HTTP dev the UI targets `https://localhost:9795`. Increment B makes the agent reachable from a browser across OSes.

**Chosen transport strategy (brainstorm 2026-05-30):** the agent serves **HTTPS on localhost with a self-signed certificate**; the user accepts it once per browser (NexU-style). Rejected for now: a public cert on a 127.0.0.1-resolving domain (zero-friction but needs domain/cert provisioning — future), and HTTP+Private-Network-Access only (Chromium-only).

## 2. Transport / mixed-content

- Agent listens on **`https://localhost:9795`** using a **self-signed cert** generated on first run (BouncyCastle, already on the agent classpath via DSS): RSA-2048, `CN=localhost`, SAN `DNS:localhost, IP:127.0.0.1`, ~10-year validity, stored in a PKCS12 keystore under a config dir (default `~/.eudss-agent/agent-keystore.p12`, password autogenerated + stored alongside or fixed-dev).
- **CORS**: the existing `EUDSS_CORS_HOSTS` allow-list (add the production site origin(s)); responses include the standard CORS headers (Javalin CORS plugin, already wired).
- **Private Network Access**: the agent answers CORS preflight with `Access-Control-Allow-Private-Network: true` so Chromium permits the HTTPS-site → localhost call (belt-and-suspenders with HTTPS).
- **Accept-once UX**: first call per browser fails (untrusted self-signed cert). The UI shows a one-time setup card linking to `https://localhost:9795/rest/health`; the user opens it, accepts the certificate, returns, reloads. Works in HTTP dev too (http page → https agent is not mixed content; only the cert needs accepting).

## 3. Agent changes (`eu-dss-agent`)

- New `AgentTls` unit: load-or-generate the self-signed PKCS12 keystore (BouncyCastle `JcaX509v3CertificateBuilder` + `JcaContentSignerBuilder`), persisted to the config dir; returns the keystore + password for the connector.
- `AgentMain.buildApp`: configure an **SSL connector** on 9795 via Javalin's Jetty hook (`cfg.jetty.modifyServer`/`addConnector` with a Jetty `SslContextFactory.Server` pointing at the keystore). Plain HTTP path kept when TLS disabled.
- Config: `EUDSS_AGENT_TLS` (default **true**; `false` = HTTP, for pure-local dev / tests). Keystore path + password via env with sane defaults.
- Add the PNA response header to the CORS/OPTIONS handling.
- Optional: expose the cert SHA-256 fingerprint in `/rest/health` so the UI can show "agent cert: …".

## 4. UI changes (`eu-dss-ui`)

- `agentApi` keeps `https://localhost:9795`. Replace the static "agent introuvable" hint with an **actionable first-run card**: a clickable link to `https://localhost:9795/rest/health` and numbered steps ("ouvrez ce lien → acceptez le certificat de l'agent → revenez et rechargez"), plus the existing "recharger" button. No contract change.

## 5. Cross-OS launch + driver

- Launch scripts: add **Windows** `bin/eu-dss-agent-windows.ps1` (and a `.bat` shim); refresh the existing macOS/Linux scripts. Each selects JDK 21, sets the OS driver default (already in `AgentConfig`: mac `/Library/SCMiddleware/libidop11.dylib`, linux `/usr/lib/libidop11.so`, win `C:\Windows\System32\idop11.dll`), and runs the jar.
- **Install & trust guide** (`docs/INSTALL.md` or README section) per OS: install JDK 21 (or use the bundled-JRE MSI on Windows), install the IDOPRO driver, run the agent, accept the cert once.

## 6. Windows MSI installer (priority demo)

Built with **`jpackage`** (bundled in JDK 21) — produces a self-contained MSI that bundles a JRE, so the end user double-clicks to install; no separate Java needed.

- **Build script** `packaging/windows/build-agent-msi.ps1`: runs `jpackage --type msi --name "EU-DSS Agent" --app-version <v> --input <staging> --main-jar eu-dss-agent-<v>.jar --main-class com.linagora.eudss.agent.AgentMain --win-console --win-menu --win-shortcut --dest <out>` (console app so the `PIN signature :` prompt works). Requires WiX 3.x on PATH.
- **CI workflow** `.github/workflows/windows-installer.yml` (`workflow_dispatch` + tags): on `windows-latest` → checkout, setup Temurin JDK 21, `mvn -pl eu-dss-agent -am -DskipTests package`, stage the jar, install WiX 3 (`choco install wixtoolset`), run the build script, `upload-artifact` the `.msi`. This produces a downloadable MSI without a local Windows machine and is the reproducible "installer creation".
- **Constraint:** the MSI can only be built on Windows. From this (macOS) repo we commit the complete build setup and **trigger + verify** the CI build via `gh workflow run` + run polling + artifact check; the MSI is installed/run on the Windows demo box by the user.

## 7. Security notes

Self-signed key lives on the user's machine; `CN=localhost` so it only authenticates the local agent. CORS + PNA restrict which web origins may call the agent. The token only ever signs digests; the PIN is entered at agent start (console). The MSI is unsigned (code-signing the MSI itself is a later/optional step).

## 8. Testing

- **Agent (automated, macOS):** a test that the keystore is generated then reloaded, and that with TLS on the agent serves HTTPS (start on a random port, `GET https://localhost:<port>/rest/health` with a trust-all `HttpClient` → 200). Existing HTTP tests run with `EUDSS_AGENT_TLS=false`.
- **Agent (macOS smoke):** start the HTTPS agent, open the UI, accept the cert, confirm the agent connects (and, with the token, the full sign flow).
- **Windows MSI:** the CI workflow builds it on `windows-latest`; we trigger it and confirm the run succeeds and the `.msi` artifact is produced. Install/run is verified by the user on Windows.
- **Windows/Linux launch scripts:** written; not executed in this session (flagged).

## 9. Out of scope (B)

Native macOS `.pkg` / Linux `.deb`/`.rpm`; code-signing the MSI; the zero-friction public-cert-on-a-domain transport (future upgrade); multi-user / hosting / auth (increment **C**).

## 10. Assumptions

- The Windows demo box has the **IDOPRO Windows driver** installed (the agent does not ship the PKCS#11 driver — same as macOS).
- JDK 21 is the runtime everywhere (the MSI bundles it; mac/linux use an installed Temurin 21).
- WiX 3.x is acceptable for `jpackage --type msi` (jpackage requires WiX 3, not 4+).
